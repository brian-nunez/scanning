# Scheduler Worker

The `scheduler-worker` is a Go service that acts as the **first stage** of the scanning pipeline.  
It continuously polls Postgres for recurring scans that are due to run, claims them, and enqueues them onto the appropriate Redis Stream for downstream processing by the `airflow-worker`.

## 1. Architecture Overview

```
┌─────────────────────────────────────────────────────────────────┐
│                       scheduler-worker                          │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  Poll ticker (default 5 s)                               │  │
│  │    └─▶ queueDepth check ──▶ claimDueScans ──▶ enqueue   │  │
│  └──────────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  Recovery ticker (default 1 m)                           │  │
│  │    └─▶ recoverExpiredClaims                              │  │
│  └──────────────────────────────────────────────────────────┘  │
└────────────────────┬────────────────┬───────────────────────────┘
                     │                │
             ┌───────▼──────┐  ┌──────▼──────┐
             │   Postgres   │  │    Redis    │
             │  (scanning)  │  │  Streams   │
             └──────────────┘  └─────────────┘
```

Postgres is the **source of truth**.  Redis Streams are the **delivery buffer** — they hold job payloads only until the `airflow-worker` acknowledges them.

## 2. Data Model

### `recurring_scans`

This table defines which assets need to be scanned and on what schedule.

| Column | Type | Description |
|---|---|---|
| `id` | UUID PK | Unique identifier for the scan definition |
| `asset_id` | UUID | The asset to scan |
| `asset_type` | TEXT | Type label for the asset (e.g. `url`) |
| `frequency` | TEXT | `daily` \| `weekly` \| `monthly` \| `manual` |
| `priority` | INT | Higher value = higher priority. Values ≥ `PRIORITY_URGENT_THRESHOLD` (default 50) go to the urgent stream |
| `status` | TEXT | `scheduled` → `claiming` → back to `scheduled` after enqueue |
| `enabled` | BOOL | Only `enabled = true` rows are considered |
| `next_run_at` | TIMESTAMPTZ | When the scan is next due. Compared to `now()` in the claim query |
| `last_run_at` | TIMESTAMPTZ | Set to `now()` after the scan is successfully enqueued |
| `spread_offset_seconds` | INT | Deterministic per-asset jitter (see [Spread Scheduling](#6-spread-scheduling)) |
| `claim_owner` | TEXT | Worker ID that holds the current claim |
| `claimed_at` | TIMESTAMPTZ | When the claim was taken |
| `claim_expires_at` | TIMESTAMPTZ | Claim TTL deadline. Used for recovery |

**Index:**
```sql
CREATE INDEX idx_recurring_scans_due
ON recurring_scans (status, next_run_at, priority DESC)
WHERE enabled = true;
```
This index is used directly by the `claimDueScans` query.

### `recurring_scan_runs`

One row per individual execution attempt.

| Column | Type | Description |
|---|---|---|
| `id` | UUID PK | Unique run ID (generated by the scheduler) |
| `recurring_scan_id` | UUID FK | Parent scan definition |
| `asset_id` | UUID | Copied from the parent scan |
| `asset_type` | TEXT | Copied from the parent scan |
| `scheduled_for` | TIMESTAMPTZ | The `next_run_at` value that triggered this run |
| `status` | TEXT | `created` → `enqueued` (set by scheduler) |
| `redis_job_key` | TEXT UNIQUE | `scan:<scan_id>:<unix_timestamp>` — used as dedupe key |
| `redis_stream` | TEXT | Stream the message was sent to (`scan_jobs:normal` or `scan_jobs:urgent`) |
| `redis_message_id` | TEXT | Stream message ID returned by `XADD` |
| `max_attempts` | INT | Maximum retries allowed (default 3, from `SCAN_MAX_ATTEMPTS`) |
| `attempt_count` | INT | Current attempt counter (incremented by `airflow-worker`) |
| `result` | JSONB | Final scan output (written by `airflow-worker`) |
| `failure_reason` | TEXT | Short failure label (written by `airflow-worker`) |
| `failure_details` | JSONB | Structured failure context (written by `airflow-worker`) |

### Redis Streams

| Stream | Purpose |
|---|---|
| `scan_jobs:urgent` | Jobs with `priority >= PRIORITY_URGENT_THRESHOLD` |
| `scan_jobs:normal` | All other jobs |

## 3. Configuration Reference

All values are read from environment variables at startup. Defaults are applied when a variable is absent or empty.

| Environment Variable | Default | Description |
|---|---|---|
| `POSTGRES_DSN` | `postgres://scanning:scanning@postgres:5432/scanning?sslmode=disable` | Postgres connection string |
| `REDIS_ADDR` | `redis:6379` | Redis host:port |
| `REDIS_PASSWORD` | _(empty)_ | Redis password |
| `REDIS_DB` | `0` | Redis logical database index |
| `REDIS_STREAM_NORMAL` | `scan_jobs:normal` | Normal-priority stream name |
| `REDIS_STREAM_URGENT` | `scan_jobs:urgent` | Urgent-priority stream name |
| `SCHEDULER_POLL_INTERVAL` | `5s` | How often to poll Postgres for due scans |
| `SCHEDULER_RECOVERY_INTERVAL` | `1m` | How often to recover expired claims |
| `SCHEDULER_CLAIM_LEASE` | `2m` | TTL for a claim before recovery resets it |
| `SCHEDULER_CLAIM_BATCH_SIZE` | `100` | Max scans claimed per poll iteration |
| `SCHEDULER_MAX_QUEUE_DEPTH` | `5000` | Back-pressure threshold: stop enqueuing when combined stream length ≥ this value |
| `SCHEDULER_REDIS_DEDUPE_TTL` | `24h` | How long the deduplication key lives in Redis |
| `SCHEDULER_WORKER_ID` | OS hostname | Used as `claim_owner` in Postgres |
| `SCAN_MAX_ATTEMPTS` | `3` | Written into each `recurring_scan_runs` row as `max_attempts` |
| `PRIORITY_URGENT_THRESHOLD` | `50` | Minimum priority value to use the urgent stream |

**Validation errors** at startup (process exits):
- `SCHEDULER_CLAIM_BATCH_SIZE` ≤ 0
- `SCHEDULER_MAX_QUEUE_DEPTH` ≤ 0
- `SCAN_MAX_ATTEMPTS` ≤ 0

## 4. Startup & Shutdown Lifecycle

```
main()
  ├─ LoadConfigFromEnv()         validate env, fail-fast if invalid
  ├─ worker.New()                create Postgres pool + Redis client
  ├─ signal.NotifyContext()      listen for SIGINT / SIGTERM
  ├─ goroutine: svc.Start(ctx)
  │     ├─ ping()                10 s timeout ping to Postgres and Redis
  │     ├─ start pollTicker      (PollInterval)
  │     ├─ start recoveryTicker  (RecoveryInterval)
  │     └─ event loop (select)
  │           ├─ ctx.Done → return nil
  │           ├─ recoveryTicker.C → recoverExpiredClaims()
  │           └─ pollTicker.C    → runOnce()
  └─ on signal / Start() error:
        svc.Stop(10 s ctx)       close Postgres pool + Redis client
```

`Start()` blocks until the context is cancelled (i.e. until shutdown signal arrives). `Stop()` is called in a 10-second timeout context to drain connections.

## 5. Main Processing Flow

Each `pollTicker` tick runs `runOnce()`:

```
runOnce()
  │
  ├─ 1. queueDepth()
  │       XLen(scan_jobs:urgent) + XLen(scan_jobs:normal)
  │       If depth >= MaxQueueDepth → log WARN "scheduler backpressure active", skip iteration
  │
  └─ 2. claimDueScans()
          SQL: SELECT … FOR UPDATE SKIP LOCKED LIMIT $ClaimBatchSize
               WHERE enabled=true AND status='scheduled' AND next_run_at <= now()
               ORDER BY priority DESC, next_run_at ASC
          UPDATE status='claiming', claim_owner=$WorkerID, claim_expires_at=now()+$ClaimLease
          → returns []recurringScan
          If 0 rows → return (nothing to do)
          │
          └─ for each scan → processClaimedScan()
```

### `processClaimedScan()` in detail

```
processClaimedScan(scan)
  │
  ├─ 1. Build redisJobKey = "scan:<scan_id>:<scheduled_for_unix>"
  │
  ├─ 2. ensureRunRow()
  │       INSERT INTO recurring_scan_runs … status='created', redis_job_key=redisJobKey
  │       ON CONFLICT (redis_job_key) DO UPDATE SET updated_at=now()
  │       RETURNING id
  │       → runID (existing or newly created)
  │
  ├─ 3. Redis SETNX "dedupe:<redisJobKey>" "1" TTL=RedisDedupeTTL
  │       If key already exists (created=false):
  │         → log INFO "dedupe key already exists, skipping redis enqueue"
  │         → advanceSchedule()
  │         → return nil  ← message will not be enqueued again
  │
  ├─ 4. NextRunAtForFrequency() → compute next scheduled run time + spread offset
  │
  ├─ 5. Select stream:
  │       priority >= PriorityUrgentThreshold → scan_jobs:urgent
  │       otherwise                           → scan_jobs:normal
  │
  ├─ 6. Build payload (see §9)
  │
  ├─ 7. redis.XAdd(stream, payload)
  │       On failure → Del(dedupeKey) to roll back guard, return error
  │
  ├─ 8. markRunEnqueued()  UPDATE status='enqueued', redis_stream, redis_message_id
  │
  └─ 9. advanceSchedule()
          UPDATE recurring_scans SET status='scheduled',
            last_run_at=now(), next_run_at=<computed>,
            spread_offset_seconds=<new_offset>,
            claim_owner=NULL, claimed_at=NULL, claim_expires_at=NULL
```

## 6. Spread Scheduling

Spread scheduling prevents a thundering-herd problem where all assets of the same frequency fire at exactly the same wall-clock second.

### Algorithm (`spread.go`)

A deterministic offset is derived from the **asset UUID**:

1. Strip hyphens from the UUID.
2. Take the first 16 hex characters and parse as a `uint64`.
3. Compute `offset = uint64 % periodSeconds`.

This ensures the same asset always lands on the same slot within its period — regardless of which worker processes it.

### `NextRunAtForFrequency(assetID, frequency, now)`

| Frequency | Period start | Period (seconds) | Resulting `next_run_at` |
|---|---|---|---|
| `daily` | Midnight UTC of the **next** calendar day | 86,400 | `midnight_tomorrow + offset_seconds` |
| `weekly` | Monday 00:00 UTC of the **next** calendar week | 604,800 | `next_monday + offset_seconds` |
| `monthly` | 1st of the **next** calendar month 00:00 UTC | days-in-next-month × 86,400 | `first_of_next_month + day_offset_days + second_offset_within_day` |
| `manual` | N/A | N/A | `now + 24h` (API-triggered; offset is always 0) |

Unsupported frequency values return an error, which causes the scan row to remain in `claiming` status until the recovery loop resets it.

## 7. Deduplication Guard

The dedupe key has the form:

```
dedupe:scan:<scan_id>:<scheduled_for_unix>
```

It lives in Redis with a TTL equal to `SCHEDULER_REDIS_DEDUPE_TTL` (default 24 h).

**Purpose:** prevent the same `(scan, scheduled_for)` pair from being enqueued more than once when multiple scheduler-worker replicas are running simultaneously or when a claim is processed twice due to recovery.

If the SETNX call indicates the key already exists, the scheduler:
1. Skips the Redis `XADD`.
2. Still calls `advanceSchedule()` so the scan's `next_run_at` advances and the scan re-enters the `scheduled` state.

If the `XADD` fails after the dedupe key was set, the dedupe key is deleted (`DEL`) so the next poll can retry.

## 8. Claim Recovery

The `recoveryTicker` (default every 1 minute) runs `recoverExpiredClaims()`:

```sql
UPDATE recurring_scans
SET status = 'scheduled',
    claim_owner = NULL,
    claimed_at  = NULL,
    claim_expires_at = NULL,
    updated_at  = now()
WHERE status = 'claiming'
  AND claim_expires_at < now();
```

Any scan stuck in `claiming` for longer than `SCHEDULER_CLAIM_LEASE` (default 2 minutes) is reset to `scheduled` and will be picked up on the next poll. This handles:

- A scheduler instance that crashed mid-claim.
- A long GC pause or network stall that prevented completion.

Each message written to a Redis Stream has the following fields:

| Field | Value |
|---|---|
| `scan_run_id` | UUID of the `recurring_scan_runs` row |

| `asset_type` | e.g. `url` |
| `priority` | Integer priority value |
| `scheduled_for` | RFC3339Nano timestamp of `next_run_at` |

## 10. Error Scenarios

### 10.1 Postgres unreachable at startup

`ping()` times out after 10 seconds → `Start()` returns an error → `main()` calls `os.Exit(1)`.

### 10.2 Redis unreachable at startup

Same as above — `ping()` checks Redis after Postgres.

### 10.3 `claimDueScans` query fails

`runOnce()` logs `ERROR "scheduler iteration failed"` and returns. The next poll ticker tick will retry. No scan rows are modified.

### 10.4 `ensureRunRow` fails (Postgres write error)

`processClaimedScan()` returns an error. The scan row remains in `claiming` status. The recovery loop will reset it to `scheduled` after `SCHEDULER_CLAIM_LEASE` expires.

### 10.5 Redis `SETNX` fails (Redis write error)

`processClaimedScan()` returns an error. The scan row stays in `claiming`. Recovery resets it.

### 10.6 `NextRunAtForFrequency` returns unsupported frequency

Returns an error before Redis XADD. The scan stays in `claiming`. Recovery resets it.

### 10.7 Redis `XADD` fails

The dedupe key is **deleted** to allow future retries. `processClaimedScan()` returns an error. The scan stays in `claiming`. Recovery resets it.

### 10.8 `markRunEnqueued` finds 0 rows updated

Returns `"mark run enqueued: run <id> not found"`. This should not happen in normal operation (the row was just inserted). The scan stays in `claiming`. Recovery resets it.

### 10.9 `advanceSchedule` finds 0 rows updated

Returns `"advance recurring scan schedule: scan <id> not found"`. This would indicate the scan definition was deleted mid-flight.

### 10.10 Back-pressure triggered

When `len(scan_jobs:urgent) + len(scan_jobs:normal) >= SCHEDULER_MAX_QUEUE_DEPTH`:

- The scheduler logs `WARN "scheduler backpressure active"`.
- No scans are claimed this iteration.
- All currently-claimed rows in `claiming` remain as-is until the recovery loop fires.

**Impact:** scans may run slightly late during sustained back-pressure periods. This is intentional — it prevents the worker from producing more work than downstream can consume.

### 10.11 Multiple scheduler-worker replicas

Replicas are safe to run in parallel because:
- `FOR UPDATE SKIP LOCKED` prevents two workers from claiming the same row.
- The Redis dedupe guard prevents duplicate enqueues even if two workers somehow claim the same row in an edge case.

### 10.12 Worker crashes mid-process (after claim, before `advanceSchedule`)

The claim lease expires and the recovery loop resets `status = 'scheduled'`. If the `recurring_scan_runs` row and/or dedupe key were already written:
- The run row remains in `created` or `enqueued` status (the `airflow-worker` may still process it if the message landed in Redis).
- The dedupe key prevents a duplicate enqueue when the recovery loop fires.

### 10.13 Graceful shutdown signal (SIGINT / SIGTERM)

`ctx.Done()` is triggered. The event loop exits the `select` cleanly. `Stop()` closes the Postgres pool and Redis client within a 10-second timeout. In-flight `processClaimedScan` calls may not complete — those scans remain in `claiming` and are recovered on next startup.
